# 小红书爬虫模块技术回顾与重构分析报告

## 1. 当前问题回顾 (Current Issues)

### 1.1 核心痛点
*   **数据准确性**: 出现“收藏/评论为0”、“发布时间未知”等现象。
*   **代码维护性**: 随着针对不同 Case (如字段错位、时间格式差异) 的修复，`stats.ts` 中的 `scrapeNoteStats` 函数已变得臃肿，逻辑分散在 API 拦截、DOM 注入脚本和后续数据合并中，呈现出典型的“补丁式”开发特征。
*   **脆弱性**: 严重依赖硬编码的 URL 匹配和特定字段映射（如 `isScrambledEndpoint`），一旦平台 API 再次变更，代码极易失效。

### 1.2 根本原因分析 (Root Cause Analysis)
1.  **API 响应的多样性与混乱**:
    *   小红书不同端点 (Endpoint) 返回的数据结构不一致。
    *   **字段语义漂移**: 例如 `/note/user/posted` 接口中，`likes` 字段实际上存的是收藏数，这种“字段挪用”现象在非公开 API 中常见，导致通用解析逻辑失效。
    *   **时间格式不统一**: 存在 Unix 时间戳 (秒/毫秒) 和 格式化字符串 (`YYYY-MM-DD HH:mm`) 混用的情况。
2.  **单一函数的职责过载**:
    *   `scrapeNoteStats` 函数同时承担了：浏览器控制、网络拦截、数据清洗、DOM 脚本注入、数据库事务等所有职责。
    *   **高耦合**: 数据解析逻辑深埋在网络回调和字符串形式的 DOM 脚本中，难以单元测试。

## 2. 优化方案：从“修补”到“重构”

为了彻底解决问题，我们需要将爬虫模块从“脚本式”升级为“工程化”架构。

### 2.1 架构设计 (Architecture Design)

将单一的 `scrapeNoteStats` 拆解为以下独立模块：

1.  **Response Parsers (解析器层)**:
    *   定义统一的 `NoteStats` 接口。
    *   实现策略模式 (Strategy Pattern)，针对不同 API URL 使用不同的解析器。
    *   *优势*: 当发现新接口或字段变动时，只需新增或修改特定 Parser，不影响主流程。

2.  **DOM Scraper (页面抓取层)**:
    *   将注入浏览器的脚本提取为独立模块。
    *   规范化 DOM 数据提取逻辑，作为 API 数据的兜底（Fallback）。

3.  **Data Normalizer (数据标准化层)**:
    *   统一处理时间格式 (`parseTime`)。
    *   统一处理数字格式 (`parseCount`)。
    *   解决字段冲突（API vs DOM 优先级策略）。

### 2.2 重构路线图 (Refactoring Roadmap)

1.  **提取标准化工具**: 创建 `utils/parsers.ts`，集中处理时间、数字解析。
2.  **模块化 API 解析**: 创建 `services/rpa/parsers/` 目录，分离不同 API 的解析逻辑。
3.  **重写主流程**: 简化 `scrapeNoteStats`，使其只负责流程编排。

---

**结论**: 我们不再在原有 `if-else` 上增加判断，而是通过重构建立一套能适应字段变化的解析体系。
